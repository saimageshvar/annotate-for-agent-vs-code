# annotate-for-agent — Design

**Date:** 2026-04-23
**Status:** Draft — pending user review

## Problem

Most code today is generated by AI coding agents. Devs oversee the agent from its terminal and use the editor to read the resulting diff. Giving structured, actionable feedback back to the agent is painful: devs copy-paste snippets, retype line references, lose context, and their feedback is unstructured prose.

## Goal

A VS Code extension that lets devs annotate code (inline, on any file or diff view) and export those annotations in an agent-friendly format. The agent consumes the export and acts on each annotation selectively.

## Non-goals

- Not a replacement for PR review tools (GitHub, GitLab).
- Not an agent invoker — the extension produces feedback payloads only.
- No multi-user or team sync. Local, per-workspace, per-dev.
- Reply threads — future scope. Single comment per annotation for now.

## High-level solution

- Native hover "+" icon on every line (both normal editor and diff view, staged + unstaged) via VS Code's `CommentController` API.
- Click "+" (or right-click → "Annotate for Agent") opens a custom **webview side pane (right)** with a category dropdown, comment textarea, Save/Cancel.
- Custom **webview side pane (left, Primary Side Bar)** lists all annotations as selectable cards with pill tags, bulk export/copy pills, filters, and status badges.
- Export formats: Markdown (default), JSON, CSV. Each includes a category legend describing dev intent (not imperative directives for the agent).
- Lifecycle: `open` → `resolved` (manual) or `stale` (auto-detected via snippet hash when code changes).

## Architecture

Three layers, all inside the VS Code extension host. No external services.

**1. Editor integration (extension host)**
- `vscode.comments.createCommentController` registered for `file://` and `git://` schemes — gives native hover "+" on every line.
- `CommentingRangeProvider` filters out removed lines in diff view (no "+" on `-` lines).
- When user clicks "+", VS Code creates a `CommentThread` via the controller. Our handler immediately disposes the thread and opens our webview in the Secondary Side Bar — native inline thread UI is never rendered.
- `TextEditorDecorationType` overlays on annotated ranges — gutter icon per status/category + left-border bar.
- `HoverProvider` on annotated ranges — hover preview card with a command-URI "Open in pane" link.
- `editor/context` menu contribution — right-click entry when selection exists.

**2. State + persistence (extension host)**
- `AnnotationStore` — in-memory map, debounced (500ms) write to `workspaceState` (VS Code memento).
- `StaleTracker` — listens to `workspace.onDidChangeTextDocument`, recomputes snippet hash for affected annotations, flags `stale` on mismatch.
- `CommentThread.range` auto-tracks line-shift from VS Code; we mirror the updated range into our stored `Annotation.range`.
- Emits events to both webviews on mutation.

**3. UI (two webviews)**
- `AnnotationListView` — Primary Side Bar. Cards, checkboxes, filters, export/copy pills.
- `AnnotationEditorView` — Secondary Side Bar. Create/edit/view modes switched by message from host.

**Message bus:** host ↔ webviews via `postMessage`. Host is single source of truth; webviews render from snapshots.

**VS Code API surface:**
- `vscode.comments.createCommentController`
- `vscode.window.registerWebviewViewProvider`
- `vscode.window.createTextEditorDecorationType`
- `vscode.languages.registerHoverProvider`
- `git` extension API (`vscode.extensions.getExtension('vscode.git').exports.getAPI(1)`) for diff hunks
- `vscode.env.clipboard.writeText`
- `vscode.workspace.fs` for export file write
- `vscode.window.showInformationMessage` for toast with action button

## Data model

```ts
type Category = 'Bug' | 'Refactor' | 'Nit' | 'Question' | 'Praise' | 'Suggestion'
type Status = 'open' | 'resolved' | 'stale'

interface Annotation {
  id: string                    // uuid
  filePath: string              // workspace-relative
  range: { startLine: number, endLine: number }  // 1-indexed, inclusive
  category: Category
  comment: string               // plain text, markdown allowed
  status: Status
  context:
    | { kind: 'file', snippet: string, snippetHash: string }
    | { kind: 'diff', before: string, after: string, hunkHeader: string, snippetHash: string }
  createdAt: number
  updatedAt: number
  exportedAt?: number
}

interface Settings {
  exportDir: string             // default: '.annotate-for-agent/exports'
  includeCodeHunk: boolean      // default: true
  defaultFormat: 'markdown' | 'json' | 'csv'   // default: 'markdown'
  autoResolveOnExport: boolean  // default: false
  includeCategoryLegend: boolean // default: true
  showResolved: boolean         // default: false
}
```

### Category metadata (single source of truth)

```ts
CATEGORY_META: Record<Category, {
  userHint: string     // UI helptext only, NEVER exported
  agentHint: string    // exported legend, describes dev's intent, no imperatives
  color: string
  priority: number
}>
```

| Category | userHint (UI only) | agentHint (exported) |
|----------|--------------------|-----------------------|
| Bug | Something is not working as intended. | Dev flagged a defect. |
| Refactor | Code works but structure could improve. | Dev suggests structural change; behavior unchanged. |
| Nit | Small preference, optional to address. | Dev notes a small preference. |
| Question | Asking for clarification, not a change. | Dev is asking, not requesting a change. |
| Praise | Positive note on good work. | Dev is affirming good work. |
| Suggestion | An alternative to consider. | Dev offers an alternative. |

Rationale: `userHint` guides dev in choosing a tag; `agentHint` conveys dev intent to the agent without imperatives that could override or conflict with the actual comment body.

### Stale detection

- On `onDidChangeTextDocument` (debounced 300ms/document): for each annotation in the changed file, recompute `SHA-1(trim(snippet))` at `range`.
- Hash mismatch → `status = 'stale'`, emit event.
- Whitespace-only changes ignored (`trim` + leading/trailing whitespace normalization).
- Line-shift tracking handled by VS Code via `CommentThread.range`; we sync our stored range from it.

### Resolution

- Manual: user clicks **Mark resolved** on card or in right pane → `status = 'resolved'`.
- Auto: if `autoResolveOnExport` is on, all exported annotations flip to `resolved` after successful export.
- **Delete** is a separate destructive action (confirm dialog).

## Editor-side flow

### Adding an annotation

1. Dev selects line(s) in editor (normal or diff view).
2. CommentController shows "+" on hover (filtered: right-side lines only in diff).
3. Click "+" → `annotateForAgent.createAnnotation` command fires.
4. Extension captures `filePath`, `range`, `context`:
   - `file` kind → current snippet + hash.
   - `diff` kind → hunk via git extension API (`diffWithHEAD`), `before` + `after` + `hunkHeader`.
5. Extension opens Secondary Side Bar and posts `init-create` to `AnnotationEditorView`.
6. Right pane renders create form:
   - Category dropdown (default `Suggestion`) + live `userHint` subtext.
   - Comment textarea, autofocus.
   - Save / Cancel CTAs.
7. Save → persist annotation, apply decoration, broadcast to list pane, switch right pane to view mode for just-created annotation.
8. Cancel → clear form, side bar stays open.

### Right-click context menu

`menus.editor/context` contribution: "Annotate for Agent", `when: editorHasSelection`. Fires same command.

### Mixed selection (unchanged + modified)

When annotating a range that spans unchanged + modified lines in diff view, export emits a unified diff hunk:
- Unchanged lines → context (space prefix)
- Added lines → `+`
- Removed lines in range → `-` (pulled from HEAD via git API)

Example:
```diff
@@ -40,6 +40,7 @@ function authUser()
 const token = getToken()
-const user = db.find(token)
+const user = await db.findAsync(token)
 return user
```

### Viewing/editing existing annotation

1. Click card in list pane → host opens file at `range` (`showTextDocument` + selection reveal).
2. Host focuses Secondary Side Bar, posts `init-view` with annotation id.
3. Right pane renders view mode: category pill, rendered comment, metadata, stale warning if applicable, actions: Edit · Mark resolved/Reopen · Delete · Copy · Export.
4. Edit → same form pre-filled, Save updates in place.

### Editing indicator

While an annotation is open in edit mode, its range gets an amplified decoration: thicker border, subtle background tint (category color, low opacity). Editor scrolls range into view. Reverts on Save/Cancel/pane close.

### Persistent clickable indicator

- Every annotated range has a persistent `TextEditorDecorationType`: gutter icon per status/category + left-border bar.
- Hover → preview card with markdown link using command URI (`command:annotateForAgent.openAnnotation?...`).
- Click link → same flow as clicking list card.
- Editor context menu also contributes "Open annotation" when cursor is inside an annotated range (keyboard-driven path).

### Hover "+" filtering in diff view

`CommentingRangeProvider.provideCommentingRanges` returns only right-side lines (added + unchanged). Removed lines excluded — they have no anchor in the current file. Documented limitation: to comment on a removal, annotate the nearest surviving line.

## Annotation list pane (Primary Side Bar)

Own view container with Activity Bar icon (speech-bubble glyph).

### Layout

```
┌────────────────────────────────────┐

├────────────────────────────────────┤
│ [x] Select all                     │
│ Filter: [Open ▾] [All cats ▾]      │
│ [ Include code hunk ] (toggle)     │
│ [Markdown ▾]  [Copy ▾]             │
├────────────────────────────────────┤
│ ┌──────────────────────────────┐   │
│ │☐ Fix null check before deref │   │

│ └──────────────────────────────┘   │
│ ┌──────────────────────────────┐   │
│ │☐ ⚠ Rename — now stale         │   │
│ │  [Nit] src/util.ts:10        │   │
│ └──────────────────────────────┘   │
└────────────────────────────────────┘
```

### Card anatomy

- Checkbox
- First line of comment (truncated with ellipsis)
- Status icon prefix if stale (`⚠`) or resolved (dimmed)
- Category pill (color from `CATEGORY_META`, tooltip = `userHint`)
- File path + line range
- Hover → highlight card, show inline action menu (⋮): Edit, Mark resolved, Delete, Copy, Export
- Click body → open in editor + right pane (view mode)

### Filters

- **Status**: Open (default) · Stale · Resolved · All
- **Category**: All (default) · multi-select (Bug, Refactor, ...)
- **Sort** (secondary): Priority (default, from `CATEGORY_META.priority`) · Newest · File path

### Bulk actions

- **Select all** selects currently visible (filtered) cards.
- **Export pill** applies to checked cards. If none checked, applies to all filtered visible cards.
- **Copy** → clipboard; **Export** → file (see naming below).

### Empty states

- No annotations: illustration + "Select code and click the + icon or right-click to add your first annotation."
- All resolved: "All clear. Toggle 'Show resolved' to see history."

### Keyboard

- Arrow keys navigate cards; Space toggles checkbox; Enter opens.
- `E` = export selected, `C` = copy selected (when pane focused).

## Export

All formats prefixed with metadata header and (if `includeCategoryLegend`) category legend using `agentHint` — never `userHint`.

### Filename

`<adjective>-<animal>-<YYYYMMDD-HHMMSS>.<ext>`

Example: `swift-otter-20260423-142305.md`. Extension derived from format.

### Default export directory

`.annotate-for-agent/exports/` in workspace root. Overridable via `annotateForAgent.exportDir` setting.

### Post-export toast

`Exported 5 annotations → swift-otter-20260423-142305.md` with **Open file** action button (`vscode.window.showInformationMessage` with item).

### Markdown format

```md
# Annotations for Agent

_Exported 2026-04-23 14:23:05 from `annotate-for-agent` · 5 open · 2 stale_

## Category legend (dev's intent)
| Category | Meaning |
|----------|---------|
| Bug | Dev flagged a defect. |
| Refactor | Dev suggests structural change; behavior unchanged. |
| ... | ... |

---

## 1. [Bug] `src/auth.ts:42-45`

**Comment:** Null-check user before toJSON. toJSON crashes on null.

```diff
@@ -42,4 +42,4 @@
- return user
+ return user.toJSON()
```
```

### JSON format

```json
{
  "exportedAt": "2026-04-23T14:23:05Z",
  "workspace": "annotate-for-agent",
  "counts": { "open": 5, "stale": 2, "resolved": 0 },
  "categoryLegend": { "Bug": "Dev flagged a defect.", "...": "..." },
  "annotations": [ /* Annotation[] minus userHint */ ]
}
```

### CSV format

Legend as leading `#`-prefixed comment rows. Main table: `path,startLine,endLine,category,status,comment,codeHunk`. Hunk inlined as escaped string.

### Toggle: Include code hunk OFF

Drops `context` / `codeHunk` fields from all formats. Payload = path + range + category + comment only.

The toggle in each webview (list pane and right pane) is two-way bound to the `annotateForAgent.includeCodeHunk` setting. Flipping it anywhere updates the setting; the setting is the single source of truth.

### Individual export

From card ⋮ menu or right pane — same formats, single-item payload. Legend still included unless disabled via setting.

### Copy action

Writes the same string the export would write to a file, but to clipboard. Never writes file.

## Testing strategy

**Unit (mocha + ts-node)**
- `AnnotationStore` CRUD + `workspaceState` roundtrip.
- Export formatters — snapshot tests per format, with/without code hunk, with/without legend.
- Slug generator — no collisions across 10k runs; output matches regex.
- Stale hash — whitespace-invariant, detects real drift.
- Category meta — guard test: `userHint` must never appear in export output.

**Integration (@vscode/test-electron)**
- Activate → create annotation → verify persistence.
- Simulate text edit → verify stale flag.
- Export → verify file written, filename matches pattern.
- Webview message handshake (host ↔ list view, host ↔ editor view).

**Manual verification checklist (pre-release)**
- Hover "+" in normal editor, staged diff, unstaged diff.
- Right-click menu when selection exists.
- Removed lines in diff → no "+".
- Category color + pill consistent across list pane, decoration, export.
- Export toast with Open link opens the file.
- Stale detection fires on hash mismatch only (whitespace-only edits don't flip status).

**CI:** Deferred.

## Performance guardrails

- Stale check debounced 300ms per document.
- Hash computation skipped if range unchanged and text length identical.
- List pane re-render diffed (keyed updates) — must stay smooth at 100+ annotations.
